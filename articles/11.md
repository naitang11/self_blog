# PostgreSQL vs MySQL：专业角度深度对比

## 前言

在选择关系型数据库时，PostgreSQL 和 MySQL 是最常被拿来对比的两个选项。两者都是开源数据库，但在架构、设计哲学和适用场景上有显著差异。本文从专业角度全面分析两者区别，帮助你在项目中做出正确选择。

---

## 一、基本概念

### 什么是 PostgreSQL

PostgreSQL 是一个高级对象关系型数据库系统，起源于 UC Berkeley 的 Ingres 项目，经过 30 多年发展已成为功能最全面的开源数据库。它支持 SQL 标准几乎全部特性，并在此基础上进行了大量扩展。

### 什么是 MySQL

MySQL 诞生于 1995 年，是最流行的开源关系型数据库之一。由于其开源免费、轻量易用的特性，LAMP 时代（Linux + Apache + MySQL + PHP）让它成为 Web 应用的事实标准。2008 年被 Sun 收购，2009 年随 Sun 被 Oracle 收购。

## 二、核心架构差异

### 1. 存储引擎

MySQL 支持多种存储引擎，插件式架构让它灵活性很高：

| 引擎 | 特点 | 适用场景 |
|------|------|---------|
| InnoDB | 默认引擎，支持事务、行级锁、外键 | OLTP 业务 |
| MyISAM | 不支持事务，查询速度快 | 读多写少 |
| Memory | 数据存内存，极快 | 临时表、缓存 |

PostgreSQL 采用单一存储引擎，但通过 MVCC（多版本并发控制）和 WAL（预写日志）实现了比 MySQL 更精细的并发控制。

### 2. 并发控制

**MySQL (InnoDB)**：
- 使用行级锁 + MVCC
- 默认隔离级别为 REPEATABLE READ
- 通过 Next-Key Lock 解决幻读问题

**PostgreSQL**：
- 使用 MVCC + 乐观并发控制
- 支持四种标准隔离级别（READ UNCOMMITTED 到 SERIALIZABLE）
- 通过 SSI（Serializable Snapshot Isolation）实现真正的可串行化隔离

### 3. 索引结构

**MySQL**：
- B+Tree（默认）
- R-Tree（空间数据）
- Hash（Memory 引擎）
- Full-text（文本搜索）

**PostgreSQL**：
- B-Tree（默认）
- Hash
- GiST（地理信息、全文检索）
- GIN（倒排索引，适合数组、全文检索）
- BRIN（块范围索引，适合大规模时序数据）
- 表达式索引和部分索引

## 三、数据类型支持

### MySQL 数据类型

相对传统，支持：
- 数值类型（INT, FLOAT, DECIMAL 等）
- 字符串类型（CHAR, VARCHAR, TEXT 等）
- 日期时间类型
- ENUM 和 SET
- JSON（5.7+ 支持，但功能有限）

### PostgreSQL 数据类型

更为丰富，支持：
- 传统类型（与 MySQL 类似）
- JSON / JSONB（完整支持，可建索引）
- 数组类型（可直接存数组）
- 范围类型（可用于日程、IP 范围）
- 几何类型（点、线、面、圆）
- 网络地址类型（INET, CIDR）
- UUID 类型
- 自定义类型（用户可自己造类型）

**PostgreSQL 的 JSONB 优势**：
```sql
-- PostgreSQL 可以对 JSON 内部字段建索引
CREATE INDEX idx_user_metadata ON users USING GIN (metadata);
SELECT * FROM users WHERE metadata->>'city' = 'Beijing';
```

## 四、SQL 标准与查询能力

### PostgreSQL 优势

**1. 窗口函数**
```sql
SELECT name, department, salary,
       RANK() OVER (PARTITION BY department ORDER BY salary DESC) as rank
FROM employees;
```

**2. CTE（公用表表达式）**
```sql
WITH monthly_sales AS (
    SELECT DATE_TRUNC('month', sale_date) as month, SUM(amount) as total
    FROM orders GROUP BY 1
)
SELECT * FROM monthly_sales WHERE total > 10000;
```

**3. 递归查询**
```sql
WITH RECURSIVE org_tree AS (
    SELECT id, name, manager_id, 1 as level
    FROM employees WHERE manager_id IS NULL
    UNION ALL
    SELECT e.id, e.name, e.manager_id, ot.level + 1
    FROM employees e, org_tree ot
    WHERE e.manager_id = ot.id
)
SELECT * FROM org_tree;
```

### MySQL 优势

MySQL 8.0 后也支持了窗口函数、CTE 和 LATERAL JOIN，但 PostgreSQL 的实现更成熟，查询优化器更智能。

## 六、性能对比

### 何时选 MySQL

- **简单 CRUD 操作**：博客、论坛、内容管理系统
- **高并发简单查询**：读多写少的场景
- **需要 replication 简单**：主从复制配置容易
- **资源受限环境**：轻量级部署

### 何时选 PostgreSQL

- **复杂查询**：多表 JOIN、子查询、聚合分析
- **大数据量**：TB 级数据处理
- **需要事务**：金融、订单等强一致性场景
- **特殊数据类型**：GIS、AI 向量、全文检索
- **未来扩展**：需要处理非结构化数据

## 八、实际选择建议

### 选 PostgreSQL 如果：

- 你需要处理复杂业务逻辑
- 项目可能往数据分析 / AI 方向发展
- 团队有 DBA 能驾驭高级特性
- 需要处理地理信息或向量数据

### 选 MySQL 如果：

- 项目是传统 Web 应用（CRUD 为主）
- 团队习惯了 LAMP/LEMP 架构
- 需要快速上手、文档丰富
- 使用 Serverless 或托管服务（成本考量）
